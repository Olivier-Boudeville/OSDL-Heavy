Hardware considerations

What should be bought

The console

First of course is the Nintendo DS itself (see a video presentation). It should be a DS Lite, which is quite superior to the previous "fat" version : better screens, thiner, better autonomy, etc. The color does not matter, but the black version is quite sober, which opens it the possibility of somewhat faking a classical PDA. This is one of the intended uses in my case, besides game programming.

As most console manufacturers enforce a unique pricing for all vendors, choose the one with the better conditions (guarantee, return policy, etc.), preferably not online : dealing with real shops is convenient, especially to have its DS replaced whenever there are dead pixels. This happens quite often apparently, and not all vendors accept easily to replace it. As for me, I bought my DS lite in France, in a FNAC shop (I could cancel my buying during a 15-day period), in may 2007, for 150 euros, and had no dead pixel.

The add-ons for homebrew development

Having a console is not enough to be able to program it : Nintendo, due to the fear of piracy, preferred to make the life of homebrewers more difficult than technically needed, notably thanks to hardened firmwares preventing modifications or execution of non-acknowledged code : apparently they are encrypted with a private key from Nintendo, the device uses the embedded public one to decipher the firmware before running it. Hence, to have access to a DS that can be programmed, one has to circumvent these protections. One of the simplest ways of doing so is to use special game-like cards, at least once. This is a vast subject and many cases have to be taken into account, see Booting Tools on Wikipedia for details.

As for me, I intended to have GNU/Linux working on my DS, and the native 4 MB of RAM where quite small, so I preferred to have both 32 MB of RAM (the maximum size that can be addressed by the DS bus) and access to mass storage on removable media. The best and simplest bet was to use in the slot 1 (the DS-specific one, small and at the top of the console, a.k.a the DS card) a SuperKey (a kind of NoPass) as a fake game used to enable the execution of code from the slot 2 (the big one at the bottom, used for GBA compatiblity, a.k.a the GBA cartridge). This slot 2 would host a SuperCard Lite (note : avoid the rumble series, it does not provide more RAM), which besides the 32 MB of RAM offers a micro-SD port.

Such removable media are not the magic bullet for the homebrewers though : to anticipate a bit on further subjects, when generating programs for the DS from your PC, to test them you may either execute them "in place", directly on your PC thanks to DS emulators (but they do not behave always as a DS would behave) or you have to send them to the DS by one way or another. This can be done thanks to the aforementioned microSD card, but it soon becomes tedious to swap again and again the card and its adapter. An alternate way would be to use a Wifi access-point or a Wifi router, preferably not a mere Wifi adapter which has to embed a specific chip etc. in order to work with the DS. Such devices could be for example the famous Linksys WRT54G routers or La Fonera, both of which are inexpensive and, on a side node, run Linux, whose firmware and software can be programmed, etc., in a very similar way to the one we are to program applications for the DS.

I bought both the SuperKey and the SuperCard Lite (black version) from FL-games for 65 euros (shipping included) and was quite happy of it (no compatibility issue between the two cards to be afraid of).

Here are the three boxes : the Nintendo DS Lite one, the SuperKey and the SuperCard. Note the two-euro coin : there are quite small indeed.


(click to enlarge)

Last be not least, one should have at least one micro-SD card, I bought a 2 GB Kingston one from PC-look for 28 euros, shipping included. My laptop had a SD reader/writer (well, Ubuntu could not use it properly but it is another story), so no real need to buy anything else. Separate USB reader/writer are quite inexpensive anyway. Check that your micro-SD card is sold with a SD adapter, so that the actual tiny micro-card can fit in a basic SD reader.

To use your micro-SD, one should first format it in FAT (i.e. FAT16, not FAT32) before use, as factory settings are not reliable in all cases, or not the expected ones. Formatting can be done both from Windows or from Linux. FAT is for the moment favored over other filesystems by most homebrewers, mainly for the sake of simplicity (there already exists a library for it on the DS, libfat) and adequate for small size storage media.

Finally in my case the full equipment cost me only 265 euros, not including the (very optional) Wifi router (a Linksys WRT54GL would cost 55 euros approximately).

What you have for the price : hardware resources

The console

Even though Nintendo designed a low-cost handheld, the DS offers quite a lot of features :

  • two 3-inch (61x46 mm, dot pitch of 0,24 mm) TFT back-lighted screens (hence the DS means Dual Screen, although Developer System is mentioned sometimes) whose maximal resolution is 256x192, with up to 260 000 colors
  • a touchpad integrated with the bottom screen. The resulting touchscreen can be used with a stylus

  • two all-purpose processors :

    • an ARM9 : ARM946E-S, main CPU, 67 MHz, between 200 and 300 MIPS, RISC 32 bit
    • an ARM7 : ARM7TDMI, co-processor, 33 MHz, about 20 MIPS, 16-bit/32-bit RISC

    The ARM (Arcos Risc Machine) family is quite widespread on embedded devices, and is known to be small, cheap and power-savvy. The Nintendo 64 offered roughly 100 MIPS (Million Instructions Per Second), PlayStation 2 and PSP (PlayStation Portable) roughly 560 MIPS, GameCube 980 MIPS. According to some sources, the DS processors are under-clocked to save some power. Some people overclocked them, which involves quite some electronical skills

  • several memory banks, notably 4 MB (4096 kB) of built-in RAM, whose layout is somewhat complex but for the most part is shared by the two processors. There is an additional 656 KB of video RAM
  • an advanced 2D rendering system and a 3D rendering system, able to offer cell-shading, fog, etc, up to 120 000 polygons rendered per second, i.e. a fill rate of about 30 millions pixels per second. The chip is adapted from the one of the Sega Dreamcast, a 120 MHz PowerVR MBX (RS version)
  • A flashable firmware, on NVRAM (Non-Volatile RAM)
  • A gamepad (four-direction keys), with 6 bigger buttons (X, Y, A, B in a cross layout; L and R on the far side of the DS) and two smaller(Start and Select)
  • integrated wireless networking, able to offer proprietary protocol (called NiFi, for Nintendo Wifi) and 802.11b Wifi connectivity, both for local exchanges between DS and communications through the Internet (range between 10 and 30 meters)
  • a 16 hardware channel Virtual Surround sound output with stereo speakers (whose quality is rather good) and standard headphone plug (3.5 mm jack). A sliding button allows to set the volume.
  • a microphone for sound input, which allows speech-recognition
  • two slots for FLASH cards : the Slot-1 (DS specific, looking a bit like a small Compact Flash card, up to 1 GB of internal storage) and the Slot-2 (legacy GBA port, used by several add-ons)
  • lithium-ion battery : in DS mode, 6 to 10 hours of play, in GBA mode up to 12 hours, in sleep mode two weeks, all in the case of a fully charge battery (4 hours of charge). Of course autonomy decreases with slot-1/slot-2 cards being added.
  • small form factor : 148,6 x 84,6 x 28,7 mm (closed) for 275g for the DS
  • built-in realtime 33MHz clock, maintaining time and date even when the DS has been shutdown
  • two green/orange/red LEDs are available, they can be seen both when the console is open or closed. By default they inform about the Wifi and the battery status

The ARM RISC architecture is heavily inspired from the GameBoy Advance one. Both have an ARM7 (but the DS one is running at 66 MHz instead of 33 MHz), explaining partly the very good retrocompatibility of DS for GBA), and the DS hardware is quite close to the one of PocketPC devices. The performances are expected to be somewhat near the Nintendo 64 ones.

Input handling can be done thanks to the traditional gamepad on the left, but also thanks to the touchscreen and/or the microphone (speaking, blowing air, etc.)

The two screens can be used also on their side, like an open book.

More infos about the DS hardware : Dev-Scene tutorial and DS Tek.

The main competitor for the DS is Sony's PlayStation Portable, PSP, although they are quite different in price, capabilities and market shares.

The Slot-1 NoPass

Apart from allowing the use of the Slot-2 devices, nothing more is given by the SuperCard and its Slot-1 counterparts. On the contrary, it consumes some power that reduces the console autonomy. A flashed firmware can be used instead.

The Slot-2 linker (a.k.a. game cart, or flash cart)

There are several slot-2 add-ons, which all provides their own set of features. From the ARM9 point of view, they can offer up to 32 MB of ROM and 64 kB of RAM.

The SuperCard Lite is one of the most interesting linkers, as it fits well in the GBA port (for DS Lite) and provides 32 MB of RAM and an access to removable storage, here a microSD card. Most of these removable cards are formatted in FAT (i.e. FAT16, not FAT32), which implies no more than 2 GB can be managed.

Programming the DS

Two general-purpose CPU to handle

For newcomers, one of the most disturbing specificities of the DS is its having multiple processors. They are expected to share the load : even both ARM are rather general-purpose, the ARM9 should take the most of computation-intensive tasks, including graphic-related operations, the ARM7 should concentrate on input management (touchscreen, most keys), wireless communications, audio output and any GameBoy emulation, whereas the 2D/3D engines will be dedicated to rendering.

The main built-in memory, consisting of one big block of 4MB memory, can be accessed by both ARM's, but one at a time. When both CPUs are trying to read main memory, one will have priority over the other.

The ARM CPU can communicate thanks to IPC (Inter-process communication) set of registers, managed thanks to a FIFO (First In, First Out) data structure.

Both CPU can trigger interrupts to each other (if the ARM9 allowed it). It is convenient to send a notification to the other CPU when for example there is data for it to be read on a shared RAM, once the calling CPU has finished filling it.

The DS ROM has to include at least two executables, one for each of the ARM. At runtime they will have to work simultaneously, which involves often having to be synchronized and to share data.

Activating only the relevant subsystems

First, the DS can run in different modes, including the GBA Mode (ARM9 not used), sleep (for the ARM7), stop, halt, etc. The DS can be waken up from various IRQ (Interrupt Request) :

  • timer triggered
  • screen being opened
  • slot-1 or slot-2 card being removed
  • certain key combination pressed (with the exception of X and Y)

Second, most subsystems of the DS can be powered independently : sound speakers, Wifi, LCD screens, the two 2D engines, the 3D rendering and geometry engines,

Selective activation is useful notably to save energy.

Miscellaneous features

The DS contains both divide and square root accelerators.

The ARM7 processor supports both 32-bit and 16-bit instructions via the ARM and Thumb instruction sets.

Various subsystems (power management, firmware, touchscreen, sound volume and control, microphone amplifier control and gain control, back-lighting of screens, power LED, battery status) can be accessed only thanks to the SPI (Serial Peripheral Interface) bus.

There are four prioritized DMA channels. They can transfer data asynchronously from the main CPU, i.e. the ARM9, if it does not access to main memory, can continue its work while the DMA transfer is going on.

There are eight incrementing 16bit timers, four for each CPU.

Most of these informations come from DS Tek.

Using some third-party software

Software distribution on the DS is mostly based on ROM images, which can be downloaded on a PC as unique files, but are read by DS software which see them as a kind of archive (a filesystem actually) containing possibly numerous files and directories.

The linker storage device issue

Before rushing for the numerous homebrew softwares available on the Internet, one must understand the problem encountered with the various linkers, the specific cards inserted to regain control of your DS.

As soon as an application has to access to a memory provided by the linker, as each linker offers its own memory interface the application has to be aware of the specific linker it will run with. Several solutions exist : one may build as many versions of one's software as supported linkers (it becomes soon tedious and messy), or one may use Chishm's DLDI, for FAT-based applications.

DLDI stands for Dynamically Linked Device Interface for libfat. The DLDI-aware software has just to be built once, and patched according to the target linker it is to run with, before being transferred to the DS. The point is that for the user it is quite easy to patch the software, and it frees the developer from having to take into account each and every linker variation on the market. Even linkers released years after one's software will be supported thanks to DLDI, with no additional efforts from the original author.

DLDI is needed whenever libfat versions more recent than Christmas 2006 are being used.

In practice, the user will need :

  • the target DLDI-enabled software to patch (of course)
  • a patching tool : the command-line or GUI dlditool, available for Windows, Linux and Mac OS X
  • the device-specific patch file which corresponds to your linker (see DLDI Device patches)

One just has to use the DLDI tool to patch the software according to its linker-specific patching specification. Another option for Windows users is to install DLDIrc, which is configured once for all with a target linker and adds to the contextual menus for each selected DS ROM (the menu obtained with the right-click) the possibility to patch it.

The extension issue

Linked to the previous subject, one can find several file extensions for Nintendo ROM, each have a specific meaning and must be chosen adequately :

  • .nds (ex: myGame.nds) : boot with the Start key
