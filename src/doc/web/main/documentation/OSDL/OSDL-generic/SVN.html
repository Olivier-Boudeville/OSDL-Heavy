OSDL's CVS policies

Overview

This document concentrates on the way OSDL uses CVS services.

Choices

Every OSDL's source file (code, but also documentation, web page and so on) is to be registered under CVS.

The reasons for their storing in control version are :

  1. avoiding loosing them : there is nothing as painful in development when sources are lost. SourceForge's servers are reliable and regularly archived. And thanks to CVS, many people who performed a checkout of our sources have their own copies.
  2. keeping all the versions available : for every file, every version can be retrieved, thus all thier history can be used, allowing roll-backs.
  3. sharing the sources : control versions systems allow users to work collaboratively on a project, in a groupware-fashion

However, as opposed to source file, no generated file (like objects [*.o, *.lo], libraries [*.a, *.la, *.so] or executables) should be versioned, since they are to be rebuilt using the sources. Furthermore their size is often significant.

At the moment, the OSDL team uses two CVS repositories :

  • the main one in SourceForge [browse]
  • the second one in our project server, which is scarcely up-to-date [browse]


Accessing our main repository

First of all, CVS shoud be set to be tunnelled thanks to SSH, otherwise one could not connect to SourceForge's servers. It is therefore necessary to set the shell variable CVS_RSH to the value "ssh" (in bash, it would be export CVS_RSH=ssh).

Any registered user for OSDL project at SourceForge is granted CVS reading/writing access. For a user whose SourceForge's name is hurricane, the following prefix should be put before any CVS command (such as ci, co, add etc.) :

cvs -z3 -d:ext:hurricane@cvs.sourceforge.net:/cvsroot/osdl

Despite the fact that all this prefix is optional when using CVS from a check-out directory, having the right alias is helpful. So, one could define in, for example, bash shell alias cvsosdl='cvs -z3 -d:ext:hurricane@cvs.sourceforge.net:/cvsroot/osdl'. Afterwards, a simple command such as cvsosdl co OSDL/OSDL-0.3 would do the trick.

Quick overview of the structure of our CVS repository

Basically, the work (development, documentation, and so on) being made by the OSDL team (including our club for game creation) respects the following structure :

  • Ceylan : our generic all-purpose C++ library
  • OSDL : our generic multimedia library
  • Tools : some tools we had to develop or improve for our own needs (such as yag-osdl)
  • Trailer : what we specifically created for our first game's trailer
  • club : some work necessary for our game creation club

Dealing with security and user-friendly approach

By default, each operation involving the SourceForge's repository triggers a SSH-handshake that leads to the asking of the user's password. As oftenly many CVS operations have to be chained, it becomes a growing pain to have to type one's password many times.

There is hopefully a quick work-around described here, which allows to export one's public key to SourceForge' servers. That way, one's machine can self-authenticate the user without any password being typed, allowing a more user-friendly CVS usage.


Some CVS-related hints

Updating one's repository

One should perform on a regular basis (i.e. at least before modifying anything under the local check-out directory) an update, which makes sure that the local files are up-to-date with regard to the CVS repository, therefore hindering users to work on an obsolete version of the file, which would result in a painful merge to be done.

To perform this update, use this command :

cvs -z3 -d:ext:hurricane@cvs.sourceforge.net:/cvsroot/osdl update -d
- or, preferably -
cvsosdl update -d

The reason for the -d option is that, without it, any newly created directory in the repository would not be copied back.

Communication

Each time a major modification of the CVS repository is being made, one should inform the other CVS users of the change, so that they will not forget to perform an update.

Another important point is to ensure that stable releases are tagged, so that at any time one can roll-back to a working set of source files. To do so recursively, from within a stable checkout directory, use :

cvs tag STABLE_yyyymmdd
for instance :
cvs tag STABLE_20040227 for 2004, February 27.

Thus, the rule of thumb might be : ensure that the recommendations of the Download page are always applicable.

CVS limitations

CVS is a useful tool, but has some defects that hopefully his replacements (ex : Subversion) won't inherit.

The most annoying drawback of CVS is that it is not possible from its clients to remove a directory on the repository. If file deleting is easy (cvs rm myFile after having removed your corresponding local checkouted file), the only way of getting rid of a directory is to log on the CVS server and to manually perform deletings in the repository directly.

As it is not possible to log on SourceForge's CVS servers, there exist only two solutions to correct a mistake involving a directory (see the import section to understand how easy it is to make mistakes) :

  • usually, do nothing : remove recursively by hand (cvs rm) all files in the directory, leaving a tree of empty directories, and wait till the next project's version, which should be cleaned up of those directories before being imported again
  • for big cases : issue a CVS project request support to SourceForge team, who will hopefully spoil a bit of their time correcting manually one's project repository


A useful reminder about cvs import

Less CVS repositories should be bungled by mistake with the import command thanks to that reminder taken from CVS' manual :

If the files you want to install in CVS reside in `wdir', and you want them to appear in the repository as `$CVSROOT/yoyodyne/rdir', you can do this :

$ cd wdir
$ cvs import -m "Imported sources" yoyodyne/rdir yoyo start

Unless you supply a log message with the `-m' flag, CVS starts an editor and prompts for a message. The string `yoyo' is a vendor tag [i.e. put you user name here], and `start' is a release tag. They may fill no purpose in this context, but since CVS requires them they must be present.

After having imported a tree, one should make vanish its root directory (ex : mv myRootDirectory myRootDirectory-putInCVS) and perform an update (cvs update -d). Then only one's will have the previously imported tree with all CVS metadata (CVS directories) copied.

After having checked the import is a success (the right content in the right place), one can safely remove completely one's private copy (rm -rf myRootDirectory-putInCVS). If something went wrong, some cvs rm black-magic will be needed.



If I had read more attentively that previous import command's description, I would not have bungled OSDL-0.1 CVS branch, forcing us to release sooner than expected an OSDL-0.2 version !
